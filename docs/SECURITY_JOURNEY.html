Fase 3: Implementasi Keamanan

3.1 HTTP Security Headers via Cloudflare Middleware

Kami membuat file middleware di Cloudflare Pages untuk menambahkan security headers ke setiap response:

functions/_middleware.js - Sebelum
export async function onRequest({ request, env }) { // Tidak ada - middleware tidak aktif return undefined; }
Masalah: Middleware tidak menambahkan header apapun ke response
functions/_middleware.js - Sesudah
const SECURITY_HEADERS = { 'Strict-Transport-Security': 'max-age=31536000; includeSubDomains; preload', 'X-Content-Type-Options': 'nosniff', 'X-Frame-Options': 'SAMEORIGIN', 'Referrer-Policy': 'strict-origin-when-cross-origin', 'Permissions-Policy': 'geolocation=(), microphone=(), camera=()', 'Content-Security-Policy': "default-src 'self'; script-src 'self' https://cdn.jsdelivr.net; ...", }; export async function onRequest(context) { const response = await context.next(); const newResponse = new Response(response.body, response); Object.entries(SECURITY_HEADERS).forEach(([key, value]) => { newResponse.headers.set(key, value); }); return newResponse; }
Solusi: Middleware sekarang mengumpulkan semua security headers dan menambahkannya ke setiap response HTTP

Penjelasan Teknis:

context.next() - Ini mengambil response dari static file yang diminta. Tanpa ini, middleware tidak bisa mendapatkan file apapun.

new Response() - Ini membuat response baru dengan body sama tapi header kosong. Kita gunakan ini agar bisa menambah header tanpa mengubah isi file.

Object.entries().forEach() - Loop melalui semua header dan tambahkan ke response satu per satu.

3.2 Content Security Policy (CSP) - Evolution

CSP adalah aturan yang memberi tahu browser sumber mana saja yang boleh dimuat.

Versi 1 (Awal)

CSP: default-src 'self';
script-src 'self' 'unsafe-inline';

Masalah: 'unsafe-inline' memungkinkan inline script yang bisa kejebol oleh XSS attack

Versi 2 (Improved)

CSP: default-src 'self';
script-src 'self' https://cdn.jsdelivr.net;
style-src 'self' 'unsafe-inline';

Solusi: Hapus 'unsafe-inline' dari script-src, biarkan inline hanya untuk style yang lebih aman

Penjelasan Teknis CSP:

Inline scripts adalah script yang ditulis langsung di HTML antara tag <script>...</script>. Kalau 'unsafe-inline' aktif, peretas bisa inject script berbahaya. Dengan 'unsafe-inline' dihapus, browser hanya menerima script dari file eksternal (js/script.js dan https://cdn.jsdelivr.net).

Versi akhir CSP kami:

Content-Security-Policy final
default-src 'self'; script-src 'self' https://cdn.jsdelivr.net; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com https://cdn.jsdelivr.net; img-src 'self' data: https:; font-src 'self' https://fonts.gstatic.com https://cdn.jsdelivr.net; connect-src 'self'; worker-src 'self' blob:; child-src 'self'; frame-ancestors 'self'

3.4 Input Validation dan Sanitasi

Setiap input dari user harus divalidasi dan dibersihkan sebelum diproses.

js/script.js - Validation Functions
function sanitizeHTML(html) { const map = { '&': '&', '<': '<', '>': '>', '"': '"', "'": ''' }; return html.replace(/[&<>"']/g, (m) => map[m]); } function validateEmail(email) { const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/; return regex.test(email); } function validatePhone(phone) { const regex = /^\d{10,12}$/; return regex.test(phone); }

Penjelasan Teknis:

sanitizeHTML - Mengubah character berbahaya (<, >, dll) jadi entity HTML. Jadi kalau user input "<script>", akan disimpan sebagai string, bukan dijalankan sebagai script.

validateEmail - Gunakan regex untuk check format valid email (harus ada @, nama, domain).

validatePhone - Check nomor telepon 10-12 digit angka saja.

Key Learning Points

1. Defense in Depth

Keamanan tidak bisa hanya di satu tempat. Kami pakai 3 layer: server headers, HTML meta tags, dan JavaScript validation. Kalau 1 layer gagal, masih ada yang lain.

2. 'unsafe-inline' adalah Enemy

Inline CSS aman aman aja, tapi inline JavaScript sangat berbahaya. Dengan 'unsafe-inline' di script-src, peretas bisa inject script di HTML dan itu akan jalan.

3. Testing Tools adalah Must-Have

SSL Labs, Security Headers, dan Mozilla Observatory memberikan feedback objektif. Grade yang kami dapat bukan dikira-kira, tapi hasil test dari tool professional.

4. Cookie Security Matters

SameSite=Strict dan HttpOnly flags mencegah cookie dicuri atau digunakan oleh website jahat. Detail kecil ini penting banget untuk CSRF protection.

5. Progressive Enhancement

Mulai dengan keamanan dasar (HTTP headers), terus refinement dan test, baru add fitur kompleks. Jangan bikin semuanya sekaligus karena bisa jadi vulnerable.
